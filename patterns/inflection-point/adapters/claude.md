# Inflection Point — Claude Code Adapter

When the user describes a situation where a major external force may be rendering their current strategy, architecture, or approach obsolete — but the evidence is ambiguous and there is emotional or institutional attachment to the status quo — apply this protocol:

## Protocol

1. **Detect the 10x force.** Help the user name the specific external change that could alter their situation by an order of magnitude. Distinguish ordinary pressure (competition, bugs, tech debt) from structural shifts (new technology making their approach obsolete, market moving to a different model, regulatory change). If it is ordinary pressure, this gem does not apply.

2. **Apply the "new CEO" test.** Ask the user: "If someone with no history in this project looked at the situation fresh, what would they do?" Surface the gap between that answer and the current course. That gap is sunk-cost bias.

3. **Assess timing.** If the user is waiting for more data before deciding, flag this: inflection points look debatable while you are in them. Waiting for certainty means deciding too late. Help the user identify what 60-70% confidence looks like and whether they are already there.

4. **Force the commitment question.** If the decision is to pivot, ensure it is total. Flag any "keep the old approach as a backup" plans as the anti-pattern (death by hedging). A half-pivot starves the new direction while preserving political competition between old and new.

5. **Define execution.** Help the user set concrete, measurable next steps aligned to the new direction. Identify where disagreement or resistance is likely and surface it now rather than letting it become passive sabotage later.

## Anti-pattern to watch for

**Death by hedging:** The user wants to pivot but also wants to keep the old approach "just in case." Flag this directly. Maintaining a fallback guarantees that neither direction gets full resources or full commitment.

## When NOT to apply

- The change is incremental, not structural (use normal problem-solving).
- The decision is easily reversible (use the two-way-door pattern instead).
- The user has not yet built the thing they would be pivoting away from (this gem is for established strategies, not greenfield decisions).
